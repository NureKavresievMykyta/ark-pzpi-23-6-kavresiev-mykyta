1 ЗАВДАННЯ
     Метою лабораторної роботи є набуття практичних навичок у формуванні концепції (Vision) та визначенні меж (Scope) програмного проєкту. Студенти мають ознайомитися з принципами опису ключових характеристик системи, її бізнес-вимог та функціональних можливостей. 
     У ході виконання завдання студент повинен провести аналіз предметної області, виявити основні проблеми, які має вирішувати майбутнє програмне забезпечення, та обґрунтувати актуальність розробки. Необхідно також здійснити моніторинг ринку існуючих рішень-аналогів, порівняти їх із власним проєктом та виділити конкурентні переваги. Окрім цього, завдання передбачає визначення цілей проєкту, опис його бізнес-логіки та архітектурної структури, що включає серверну частину, клієнтські веб- та мобільні додатки, а також IoT-компоненти. Для кожної з цих складових потрібно сформулювати чіткі функціональні вимоги. 
     Важливою частиною роботи є вибір технологічного стеку, ресурсів та інструментів розробки. Результатом роботи має стати оформлений документ Vision & Scope, який містить повний опис проблеми, концепцію рішення, деталізацію функціоналу та технічні аспекти реалізації. Підготовлений документ необхідно експортувати у формат PDF для завантаження на навчальну платформу, а також зберегти як текстовий файл у кодуванні UTF-8 для розміщення в особистому репозиторії на GitHub. Під час захисту роботи студент має вміти пояснити архітектурні рішення, бізнес-цілі та технічну основу проєкту.



2 ОПИС ВИКОНАНОЇ РОБОТИ
     У процесі виконання лабораторної роботи було розроблено документ Vision & Scope для спеціалізованої інформаційної системи, призначеної для автоматизації процесів в офтальмологічних клініках. Робота включала аналіз поточної ситуації в галузі, визначення бізнес-можливостей, формування бачення продукту, а також детальний опис функціональності та технічного середовища. Першим етапом став аналіз проблематики. 
     Було виявлено, що основною проблемою сучасних клінік є "інформаційні розриви" між діагностичним обладнанням та історією хвороби пацієнта, а також ризики, пов'язані з паперовим документообігом та ручним введенням складних оптичних параметрів. Це підтвердило необхідність створення системи, яка б автоматично збирала, структурувала та зберігала медичні дані. На другому етапі було проведено порівняння з аналогами, такими як універсальні МІС ("Helsi", "Doctor Eleks") та спеціалізоване ПЗ від виробників обладнання (Zeiss, Topcon). Аналіз показав, що перші не мають специфічних офтальмологічних модулів, а другі є занадто дорогими та закритими. Це дозволило сформулювати унікальну пропозицію нашого проєкту: створення гнучкої системи з нішевою спеціалізацією, яка враховує українські стандарти та інтегрується з різним обладнанням. 
     Наступним кроком стало формування бачення рішення (Vision). Головна мета системи - цифрова трансформація клініки, де кожна маніпуляція фіксується автоматично. Було визначено ключові функціональні модулі (MF-1 - MF-7), серед яких: електронна медкартка зі специфічними полями для офтальмології, модуль інтеграції з обладнанням, конструктор рецептів, система логістики та аналітики. Далі було окреслено рамки початкового релізу (Scope). Він включає серверну частину для обробки бізнес-логіки, IoT-клієнт для зчитування даних з приладів, веб-інтерфейс для адміністраторів та мобільний додаток для лікарів. 
     Також спроєктовано функції для майбутніх версій, такі як кабінет пацієнта, телемедицина та інтеграція з eHealth. Завершальним етапом стало технічне описування середовища. В якості серверної платформи обрано Java Spring Boot для надійності та безпеки, база даних реалізована на PostgreSQL. Веб-клієнт базується на React.js з використанням Tailwind CSS, а мобільний додаток розробляється на Android Java. Для розгортання системи передбачено використання Docker-контейнеризації. 
     В результаті було створено комплексний документ Vision & Scope, який повністю описує концепцію системи управління офтальмологічною клінікою, від бізнес-вимог до технічної реалізації.


3 ВИСНОВКИ
     Під час виконання лабораторної роботи було створено документ типу Vision & Scope для програмного проєкту, спрямованого на автоматизацію діяльності офтальмологічних клінік. У ході роботи було поглиблено знання щодо етапів життєвого циклу проєкту, зокрема формування вимог та визначення меж системи. Було проведено детальний аналіз предметної області, що дозволило виявити критичні недоліки існуючих підходів до ведення медичної документації в офтальмології. Це допомогло чітко сформулювати конкурентні переваги проєкту, такі як автоматичний імпорт даних з діагностичних приладів, спеціалізовані інтерфейси для рецептів та адаптація під локальний ринок. 
     Особливу увагу було приділено проєктуванню архітектури системи. Було визначено ролі основних компонентів: серверної частини на Java Spring Boot як ядра системи, IoT-модуля для взаємодії з апаратурою, а також клієнтських додатків для різних груп користувачів. Набуто навичок вибору технологічного стеку, який забезпечує надійність, масштабованість та безпеку медичних даних (PostgreSQL, HTTPS шифрування, Docker). Також було опрацьовано структуру документації згідно з міжнародними стандартами, включаючи розділи бізнес-вимог, бачення рішення, рамок проєкту та бізнес-контексту. Це дозволило сформувати цілісне уявлення про майбутній продукт та план його реалізації. 
     Таким чином, виконання роботи дозволило закріпити теоретичні знання та отримати практичний досвід у створенні професійної технічної документації для складних програмних систем.

1 BUSINESS REQUIREMENTS (БІЗНЕС-ВИМОГИ)
1.1 Background (Передумови та аналіз проблеми)
     Сучасний стан офтальмологічної галузі характеризується стрімким розвитком діагностичних технологій, проте рівень цифровізації внутрішніх процесів у багатьох клініках залишається недостатнім. Основна специфіка офтальмології полягає у необхідності обробки та зберігання величезної кількості прецизійних параметрів, таких як показники рефракції, кривизни рогівки, внутрішньоочного тиску, товщини сітківки та результатів комп’ютерної периметрії. На сьогоднішній день більшість медичних закладів стикаються з критичною проблемою "інформаційних розривів", коли дані з різних діагностичних приладів існують автономно, не будучи інтегрованими в єдину історію хвороби пацієнта.
     Використання традиційних паперових носіїв або застарілих загальних систем електронного обліку створює низку суттєвих перешкод. По-перше, це високий ризик втрати або пошкодження медичної документації, що є неприпустимим при веденні хронічних пацієнтів, наприклад, з глаукомою або діабетичною ретинопатією, де моніторинг стану зору у динаміці протягом років є життєво необхідним. По-друге, людський фактор при ручному перенесенні складних числових даних з чеків авторефрактометрів до карток пацієнтів призводить до помилок, які можуть критично вплинути на правильність підбору корекції або призначення хірургічного втручання.
     Крім того, відсутність централізованої системи унеможливлює швидку аналітичну обробку даних. Керівництво клінік не має інструментів для оперативного відстеження ефективності роботи персоналу, завантаженості діагностичних кабінетів та аналізу структури захворюваності серед пацієнтів. Процес запису на прийом та координації пацієнтів між різними фахівцями
     (наприклад, від лікаря-діагноста до хірурга-офтальмолога) часто є неоптимізованим, що створює черги та знижує загальну пропускну здатність медичного центру. Таким чином, розробка спеціалізованої програмної системи є необхідним кроком для забезпечення якості медичних послуг та оптимізації витрат клініки.
1.2 Business Opportunities (Бізнес-можливості та аналіз аналогів)
     Аналіз ринку програмного забезпечення для медицини показує, що існуючі рішення можна розділити на дві категорії: універсальні медичні інформаційні системи (МІС) та спеціалізоване закордонне програмне забезпечення від виробників обладнання. Універсальні МІС, такі як "Doctor Eleks" або "Helsi", хоча і мають широкий функціонал для загальної практики, часто не містять специфічних офтальмологічних модулів. У таких системах відсутні зручні інтерфейси для заповнення формул рецептів на окуляри, графічні редактори для позначення патологій на схемі очного дна та можливості автоматичного імпорту даних з вузькопрофільних приладів.
     З іншого боку, програмні комплекси від таких гігантів, як Zeiss або Topcon, є надзвичайно дорогими та часто закритими для інтеграції з обладнанням інших брендів. Це створює монопольну залежність клініки від одного виробника, що є економічно недоцільним для середніх та малих приватних закладів. Наша система пропонує вирішення цієї проблеми шляхом створення гнучкого, універсального продукту, який орієнтований саме на український ринок та специфіку роботи локальних клінік.
     Перспективи та переваги проєкту:
- нішева спеціалізація: продукт розробляється з урахуванням стандартів офтальмологічного огляду, що включає специфічні поля

для параметрів OD (праве око) та OS (ліве око), автогенерацію рецептів та протоколів лікування;
- економічна ефективність: модель монетизації передбачає як щомісячну передплату (SaaS) для невеликих кабінетів, так і продаж повноцінних ліцензій для великих мереж. Це дозволяє охопити широкий сегмент ринку;
- інтеграційний потенціал: використання сучасного стеку технологій (Java, PostgreSQL, React) дозволяє легко інтегрувати систему з державними реєстрами (eHealth) та внутрішніми бухгалтерськими системами клініки;
- масштабованість: можливість розширення функціоналу за рахунок модулів телемедицини, де пацієнти зможуть отримувати консультації та результати оглядів через мобільний додаток, що відкриває нові канали доходу для клініки.
     Ринок приватної офтальмології в Україні демонструє стабільний ріст, а вимога держави щодо обов'язкового ведення електронної медичної документації робить впровадження подібних систем неминучим. Отже, даний проєкт має високий комерційний потенціал та соціальну значущість, забезпечуючи перехід від застарілих методів управління до цифрових стандартів охорони здоров'я.

2 VISION OF THE SOLUTION (ОКРЕСЛЕННЯ КОНЦЕПЦІЇ)
     2.1 Vision Statement (Окреслення концепції)
     Програмна система для офтальмологічних клінік розробляється як комплексне інтегроване рішення, що має на меті повну цифрову трансформацію робочих процесів у спеціалізованих медичних закладах. Кінцева концепція проєкту передбачає створення середовища, у якому медична інформація стає доступною, структурованою та захищеною, що дозволяє лікарю зосередитися на клінічній роботі.
     У світі, де впроваджена ця система, кожна офтальмологічна маніпуляція - від вимірювання внутрішньоочного тиску до складних мікрохірургічних втручань - автоматично фіксується у цифровій історії хвороби. Система виступає як інтелектуальний помічник, що автоматично порівнює поточні показники пацієнта з його попередніми результатами, візуалізує динаміку змін у вигляді графіків та сигналізує про відхилення від норми. Це забезпечує якісно новий рівень діагностики, дозволяючи виявляти патології на ранніх стадіях. Для пацієнта це означає отримання прозорого сервісу, де всі рецепти, знімки та призначення доступні в електронному вигляді, а ризик помилки через нерозбірливий почерк або втрачену картку повністю нівельовано.
     2.2 Major Features (Головна функціональність)
     Нижче наведено детальний перелік основних функціональних модулів системи, які забезпечують її високу ефективність та відрізняють від універсальних рішень:
- MF-1: інтелектуальна електронна медична картка офтальмологічного пацієнта. Система забезпечує ведення спеціалізованої історії хвороби,

яка включає не лише текстові записи, а й структуровані дані за стандартами офтальмологічного огляду. Сюди входить детальна фіксація стану переднього та заднього відрізків ока, можливість анотування завантажених зображень (фото фундус-камери, ОКТ) безпосередньо у додатку та автоматичне формування анамнезу на основі попередніх візитів;
- MF-2: модуль автоматизованої діагностики та інтеграції з обладнанням. Це ядро системи, яке дозволяє отримувати дані з авторефрактометрів, пневмотонометрів та іншого діагностичного обладнання. Система автоматично розподіляє отримані числові значення за відповідними полями для правого (OD) та лівого (OS) ока, виключаючи необхідність ручного введення та мінімізуючи ризик механічних помилок;
- MF-3: система керування графіком прийомів та логістикою клініки. Функціонал включає інтерактивний календар для адміністраторів та лікарів, який дозволяє не лише записувати пацієнтів, а й керувати завантаженістю конкретних діагностичних кабінетів. Система підтримує різні типи візитів (первинний огляд, післяопераційний контроль, консультація вузького спеціаліста), для кожного з яких автоматично підбирається необхідний часовий слот;
- MF-4: електронний конструктор офтальмологічних рецептів. Модуль дозволяє миттєво генерувати рецепти на окуляри будь-якої складності (стигматичні, астигматичні, прогресивні) та контактні лінзи. Програма автоматично перевіряє рецепт на відповідність діагностичним даним та формує друковану форму згідно з державними стандартами, включаючи необхідні параметри, такі як міжзінична відстань (Dp) та вертексна відстань;

- MF-5: аналітичний модуль та фінансова звітність. Система акумулює дані про надані послуги, використані матеріали та фінансові надходження. Керівник клініки отримує доступ до звітів у реальному часі: кількість прийнятих пацієнтів, середній чек, ефективність кожного лікаря та динаміку записів. Це дозволяє приймати обґрунтовані управлінські рішення на основі реальних цифр;
- MF-6: модуль безпеки та розмежування доступу. Оскільки медичні дані є надзвичайно чутливими, система забезпечує багаторівневу авторизацію. Адміністратори мають доступ до фінансів та розкладу, лікарі - до медичних карток, а молодший медичний персонал - лише до виконання призначених маніпуляцій. Усі зміни в системі логуються, що дозволяє відстежити, хто і коли вносив правки у дані пацієнта;
- MF-7: система автоматичних сповіщень та взаємодії з пацієнтами. Автоматизація розсилки нагадувань про запланований візит, сповіщень про готовність результатів аналізів або необхідність планового щорічного огляду. Це значно підвищує лояльність клієнтів та зменшує кількість пропущених прийомів.

3 SCOPE AND LIMITATIONS (РАМКИ ТА ОБМЕЖЕННЯ)
     3.1 Scope of Initial Release (Рамки первинного випуску)
     Перший випуск програмної системи зосереджений на створенні стабільного ядра для автоматизації ключових процесів офтальмологічної клініки. Основна увага приділяється надійності зберігання даних, швидкості доступу до медичних карток та забезпеченню базової взаємодії між усіма компонентами системи.
     а) Функціональність серверної частини (Java Spring Boot): Серверна частина є центральним вузлом системи, який забезпечує обробку всієї бізнес-логіки та безпечну взаємодію з базою даних PostgreSQL. У першому релізі реалізуються такі компоненти:
1. Модуль керування доступом: Впровадження системи авторизації та аутентифікації на основі ролей. Сервер забезпечує розмежування прав між адміністраторами, лікарями та допоміжним персоналом, гарантуючи, що конфіденційна медична інформація доступна лише авторизованим фахівцям;
2. Централізоване сховище даних: організація реляційної структури в PostgreSQL для зберігання персональних даних пацієнтів, детальних результатів обстежень, історії візитів та сформованих рецептів. Використання механізмів транзакційності для забезпечення цілісності даних;
3. Rest api: розробка набору програмних інтерфейсів для обміну даними у форматі JSON з веб-додатком та мобільним клієнтом;
4. Система аудиту та логування: фіксація всіх критичних дій користувачів (створення, редагування або видалення записів) для забезпечення прозорості та відповідності законодавчим вимогам щодо обробки медичної інформації.
     б) Функціональність IoT-клієнта: IoT-компонент у першому релізі виконує роль інтелектуального шлюзу між діагностичним обладнанням та центральним сервером:
1. Модуль зчитування даних: реалізація драйверів для взаємодії з авторефрактометрами та пневмотонометрами через стандартні інтерфейси зв’язку;
2. Парсер медичних протоколів: автоматичне розпізнавання вихідних текстових або бінарних даних пристрою, виділення ключових параметрів зору (сфера, циліндр, вісь, показники тиску) та їх перетворення у структурований формат для передачі на сервер;
3. Буферизація повідомлень: забезпечення локального зберігання результатів обстеження у разі тимчасової відсутності мережевого зв’язку із сервером.
     в) Функціональність веб-клієнта (React.js + Tailwind CSS): Веб-інтерфейс орієнтований на роботу адміністраторів та реєстратури, а також на перегляд розширеної аналітики керівництвом:
1. Модуль реєстрації та розкладу: гнучка сітка графіку прийомів, можливість швидкого створення картки нового пацієнта, керування статусами візитів та чергою;
2. Кабінет головного лікаря: панель моніторингу основних показників діяльності клініки, перегляд зведеної статистики за обраний період та формування звітів для друку;
3. Управління довідниками: налаштування переліку послуг, спеціалізацій лікарів та кабінетів клініки;
     г) функціональність мобільного клієнта (Android Java): мобільний додаток розроблений як персональний асистент пацієнта, що забезпечує прозорість лікувального процесу та швидкий доступ до медичних сервісів:
1. цифровий профіль здоров'я: інтуїтивно зрозумілий інтерфейс для перегляду результатів огляду, де параметри для правого (OD) та лівого (OS) ока відображаються у зручній для сприйняття формі з поясненнями основних показників;
2. доступ до діагностичного архіву: можливість перегляду завантажених лікарем знімків ока та результатів обстежень безпосередньо зі смартфона, що дозволяє пацієнту мати при собі повну історію хвороби в електронному вигляді;
3. інтерактивна взаємодія з клінікою: перегляд історії відвідувань у реальному часі, контроль поточних призначень та отримання електронних рецептів на окуляри чи лінзи з можливістю їх збереження як PDF-документів для подальшого використання.
     3.2 Scope of Subsequent Releases (Рамки наступних випусків)
     У майбутніх ітераціях проєкту планується розширення можливостей системи для підвищення рівня автоматизації та залученості пацієнтів:
- клієнтський мобільний додаток для пацієнтів: створення окремого інтерфейсу, де пацієнти зможуть самостійно записуватися на прийом, переглядати свої рецепти та отримувати результати обстежень у зручному форматі;
- телемедичний модуль: реалізація функціоналу для проведення онлайн-консультацій за допомогою захищеного відеозв’язку;
- інтеграція з державними системами (eHealth): забезпечення повної синхронізації даних з центральною базою даних електронної системи охорони здоров’я України для автоматичного формування електронних направлень та рецептів за програмою "Доступні ліки";
- запуск особистого кабінету пацієнта для онлайн-запису. Це дозволить пацієнтам самостійно обирати зручний час для прийому, бачити актуальний графік лікарів та отримувати підтвердження запису в режимі реального часу;
- інтеграцію з хмарними сховищами для збереження великих обсягів діагностичних зображень. Для оптимізації навантаження на локальні сервери клініки буде реалізовано механізм автоматичного вивантаження цих даних у захищені хмарні сховища;
- впровадження модулів підтримки прийняття клінічних рішень на базі накопиченої статистики. На основі накопичених деперсоніфікованих даних система зможе пропонувати лікарю попередні висновки, порівнюючи поточні показники пацієнта з еталонними значеннями та типовими клінічними сценаріями.

4 BUSINESS CONTEXT (БІЗНЕС-КОНТЕКСТ)
     4.3 Operating Environment (Робоче середовище)
     Для успішного функціонування програмної системи та забезпечення високої швидкості обробки медичних даних було обрано сучасний технологічний стек, який відповідає промисловим стандартам розробки. Кожен компонент системи підібраний з урахуванням потреби у масштабованості, безпеці та кросплатформності.
     а) Програмні платформи та мови програмування:
1. Серверна частина (Backend): реалізується на мові програмування Java із використанням фреймворку Spring Boot. Цей вибір обумовлений високою стабільністю мови, наявністю потужних засобів для побудови RESTful API та розвиненою екосистемою для забезпечення безпеки даних. Spring Security використовуватиметься для керування сесіями та авторизації користувачів;
2. Веб-клієнт (Frontend): розробляється на базі бібліотеки React.js. Це забезпечує створення динамічного та швидкого інтерфейсу, що критично важливо для роботи реєстратури клініки. Для стилізації використовується CSS-фреймворк Tailwind CSS, який дозволяє створити адаптивний та лаконічний дизайн, зручний для використання на моніторах з різною роздільною здатністю; 
3. Мобільний клієнт: розробляється в середовищі Android Studio на мові Java. Це дозволяє забезпечити прямий доступ до апаратних засобів пристрою (камера, Bluetooth, внутрішня пам'ять) та гарантує стабільну роботу на планшетах і смартфонах лікарів; 
4. Система управління базами даних (СУБД): використовується PostgreSQL. Вона обрана через свою надійність, підтримку складних реляційних зв'язків та можливість ефективної роботи з великими масивами текстових і числових даних. PostgreSQL дозволяє реалізувати складні запити для формування медичної аналітики без втрати продуктивності.
     б) Інструменти розробки та розгортання:
1. Керування версіями: весь сирцевий код проекту зберігається та обмінюється через систему Git із використанням платформи GitHub. Це забезпечує прозорість розробки, можливість відкату до попередніх версій та спільну роботу над модулями; 
2. Середовища розробки (IDE): Eclipse IDE для серверної частини, Android Studio для мобільного додатка та Visual Studio для веб-інтерфейсу; 
3. Контейнеризація: для спрощення розгортання системи на серверах клініки або у хмарі передбачено використання Docker, що дозволяє ізолювати компоненти системи та уникнути конфліктів середовища.
     в) Вимоги до доступності, надійності та цілісності. Оскільки система оперує критично важливими медичними даними, до неї висуваються підвищені експлуатаційні вимоги:
1. Продуктивність: час відповіді сервера на запити клієнтських додатків при отриманні списку пацієнтів не повинен перевищувати 500 мс. Обробка та візуалізація графічних даних (знімків ока) має відбуватися без затримок, що заважають роботі лікаря; 
2. Цілісність та захист даних: усі персональні дані пацієнтів повинні шифруватися при зберіганні у базі даних. Передача даних між клієнтами та сервером здійснюється виключно через захищений протокол HTTPS з використанням SSL-сертифікатів; 
3. Робоче оточення користувачів: веб-клієнт має бути сумісним з останніми версіями браузерів Google Chrome, Mozilla Firefox та Microsoft Edge. Мобільний додаток повинен підтримувати пристрої на базі Android версії 8.0 та вище.
