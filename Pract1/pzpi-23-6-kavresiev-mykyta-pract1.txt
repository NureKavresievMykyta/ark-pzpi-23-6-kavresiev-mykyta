2 ЗАВДАННЯ
     Метою практичного заняття є ознайомлення з принципами написання чистого, ефективного та підтримуваного коду. Необхідно обрати мову програмування (у даному випадку Perl), дослідити основні рекомендації щодо стилю коду, іменування змінних, структурування програм та рефакторингу. Студент повинен продемонструвати ці правила на конкретних прикладах, пояснити важливість їх дотримання та вплив на якість програмного продукту.
     Окремим завданням є аналіз інструментів для автоматичної перевірки якості коду та підготовка матеріалів для презентації результатів дослідження. Результатом роботи є оформлений звіт, що містить теоретичні відомості та практичні приклади.
3 ОПИС ВИКОНАНОЇ РОБОТИ
     3.1 Правила оформлення програмного коду на Perl
     Визначено тему роботи - дослідження стандартів оформлення коду мовою Perl. Основою дослідження стали офіційна документація perlstyle (man perlstyle) та книга Деміана Конвея "Perl Best Practices". Головна ідея Perl - "There Is More Than One Way To Do It" (Є більше одного способу зробити це), однак для командної розробки критично важливо дотримуватися суворих обмежень, щоб код не перетворився на нечитабельний набір символів.
     3.2 Фундамент надійності: Прагми strict та warnings
     Робота розпочалася з аналізу найважливішого правила в Perl. Визначено, що будь-який скрипт повинен починатися з підключення прагм strict та warnings. Це змушує розробника явно оголошувати змінні та попереджає про потенційні логічні помилки ще до запуску програми.
Приклад:
# Погано
$x = 10; # Змінна не оголошена, глобальна область видимості

# Добре
use strict;
use warnings;
my $x = 10; # Лексична змінна

     3.3 Форматування коду: Відступи та блоки
     Досліджено правила візуального оформлення. Встановлено стандарт відступу у 4 пробіли (табуляція не рекомендується через різне відображення в редакторах). Для фігурних дужок прийнято використовувати стиль K&R ("єгипетські дужки"), де відкриваюча дужка залишається на тому ж рядку, що й оператор.
Приклад:
# Добре
if ($condition) {
    action();
} else {
    other_action();
}

     3.4 Конвенції іменування змінних та підпрограм
     Систематизовано правила іменування. В Perl історично склався стиль snake_case (маленькі літери з підкресленням) для змінних та імен функцій. Для імен модулів (класів) використовується PascalCase. Константи прийнято писати у SCREAMING_SNAKE_CASE.
Приклад:
# Змінні та функції
my $user_count = 0;
sub calculate_total { ... }

# Константи
use constant MAX_RETRIES => 5;

# Модулі
package My::App::User;
     3.5 Постфіксні умови та спрощення логіки
     Проаналізовано унікальну особливість Perl - постфіксні умови (statement modifiers). Це дозволяє уникати глибокої вкладеності (nested if) та робити код більш схожим на людську мову. Рекомендація полягає у використанні постфіксних форм для коротких виразів, що покращує читабельність.
Приклад:
# Погано (надлишкова вкладеність)
if ($is_error) {
    die "Error occurred";
}

# Добре (чистий код)
die "Error occurred" if $is_error;
# або
return unless defined $result;

     3.6 Робота зі списками: map та grep
Досліджено функціональні можливості мови. Замість громіздких циклів foreach для трансформації або фільтрації масивів рекомендовано використовувати вбудовані функції map та grep. Це робить код декларативним та компактним.
Приклад:
# Погано
my @squared;
foreach my $num (@numbers) {
    push @squared, $num * $num;
}

# Добре
my @squared = map { $_ * $_ } @numbers;
     3.7 Підпрограми та розпакування аргументів
     Визначено правило роботи з аргументами функцій. В Perl аргументи передаються через масив @. Рекомендовано завжди явно "розпаковувати" їх у змінні на початку функції, а не звертатися за індексом $[0]. Це робить сигнатуру функції зрозумілою для інших розробників.
Приклад:
sub connect_db {
    my ($host, $port) = @_; # Явне іменування аргументів
    # ...
}
     3.8 Документування коду (POD)
     Розглянуто стандарт документування POD (Plain Old Documentation). Коментарі мають бути вбудовані прямо в код і описувати призначення модулів та методів, а не очевидні дії. Це дозволяє генерувати HTML-документацію автоматично.
Приклад:
=head1 NAME
My::Module - Модуль для обробки даних
=head1 SYNOPSIS
use My::Module;
...
=cut
3.9 Інструменти контролю якості: PerlCritic та PerlTidy
Виконано огляд інструментів автоматизації.
- PerlTidy - автоматично форматує код згідно зі стандартами (розставляє відступи, пробіли, дужки);
- PerlCritic - статичний аналізатор, який перевіряє код на відповідність рекомендаціям книги "Perl Best Practices" та знаходить потенційно небезпечні конструкції.
     3.10 Висновки щодо розділу
     Підсумовано, що Perl є дуже гнучкою мовою, що створює ризик написання заплутаного коду ("write-only"). Дотримання описаних вище правил перетворює Perl-код на структурований, читабельний та легкий у підтримці продукт.
4 ВИСНОВКИ
     У ході виконання практичного завдання було опрацьовано основні правила оформлення програмного коду на мові Perl відповідно до офіційного посібника perlstyle та стандартів спільноти. Під час роботи було проаналізовано, як суворий стиль кодування впливає на життєвий цикл програмного забезпечення.
     Було отримано навички використання прагм безпеки (strict, warnings), правильного структурування блоків коду та застосування ідіоматичних конструкцій Perl (map, grep, постфіксні умови) для зменшення візуального шуму. Окрему увагу приділено іменуванню змінних у стилі snake_case та документуванню через POD.
     Також розглянуто інструменти статичного аналізу (PerlCritic), які дозволяють автоматизувати процес Code Review. Результати роботи показують, що навіть для такої гнучкої мови, як Perl, існують чіткі стандарти, дотримання яких є обов'язковим для професійної розробки. Це дозволяє уникнути типових помилок, спростити відлагодження та полегшити введення нових розробників у проект.
